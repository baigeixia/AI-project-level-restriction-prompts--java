# 修改代码控制流程

## 修改前（必须执行）

1. 读取类 `index.json` → 获取当前方法签名快照（返回值、参数、注解、调用链）
2. 读取类 `SKILL.md` → 确认本次修改必须遵守的类级约束
3. 确认影响范围：查 index.json 中 `calledBy` 字段，确认调用方列表
4. 如影响超过 2 个调用方 → 警告用户，等待确认后再执行

## 修改中（强制规则）

- 只修改用户指定的方法，不顺带调整无关代码（防止上下文污染）
- 修改方法签名（参数/返回值）前必须告知影响的调用链，并评估是否向下兼容
- 新增参数必须向下兼容（提供默认值或方法重载）
- 修改涉及事务的方法：确认 @Transactional 传播行为是否需要调整
- 修改涉及并发的方法：确认锁的粒度和加锁顺序是否仍然正确
- 修改 VO 的 Long 类型 ID 字段：确认是否加了 @JsonSerialize 防精度丢失

## 修改后（必须执行）

1. 更新该方法在类 `index.json` 中的描述（参数、返回值、功能、callChain）
2. 方法删除 → index.json 标记 `"status": "deprecated"`，注明 `"replacedBy"` 替代方法名，源码加 @Deprecated
3. 方法新增 → 追加到 index.json 的 `methods` 数组
4. 如涉及数据库字段变更 → 提示用户同步更新 DO 类和 resultMap

## 禁止的修改模式

- 禁止修改 Service 时顺带重构 Controller
- 禁止未经确认删除方法（可能破坏调用方）
- 禁止单次对话修改超过 5 个方法
- 禁止将业务逻辑从 Service 下沉到 Mapper
