# 代码审查流程（基于阿里规范）

## 审查维度

### 1. 规范符合性（阿里规范强制项）
- 命名是否规范（类/方法/变量/常量/包/数据库字段）
- 是否存在魔法值
- POJO 类是否规范（toString/包装类型/无 is 前缀/无默认值）
- 异常处理是否规范（自定义异常/不空 catch/不用异常做流程控制）
- 日志是否规范（SLF4J/@Slf4j/占位符/不打敏感信息）
- 并发是否规范（线程池/ThreadLocal 回收/锁粒度）

### 2. 分层合规性
- Controller 是否有业务逻辑泄漏
- Service 是否直接操作 DAO 以外的层
- Mapper 是否包含业务判断

### 3. 安全性审查
- 是否有 SQL 注入风险（${}）
- 是否有未做脱敏的敏感字段日志
- 是否缺少水平权限校验
- 写接口是否缺少防重提交

### 4. 数据库操作审查
- 是否有 SELECT *
- 是否有超过 3 表的 join
- 是否有左/全模糊查询
- 更新操作是否同步更新 update_time
- 是否有不必要的全字段 update（只更新变动字段）

### 5. 完整性审查
- 新增方法是否在 index.json 中更新
- VO 的 Long 类型 ID 是否转 String
- 分页接口是否用 AjaxPage<T>

## 输出格式

```
🔍 代码审查报告 — [ClassName]
──────────────────────────────
✅ 通过：[规则描述]
⚠️  建议：[问题描述] → [改进建议（对应阿里规范条目）]
❌ 违规：[阿里规范条目（强制）] → [具体问题描述] → [修改方案]
```
